## 키워드

- 소프트웨어 생명 주기 모형
    - 폭포수 모형 : 전통적, 선형 순차적
    - 프로토타입 모형 : 새로운 요구사항 적용, 비효율적 언어, 알고리즘 사용
        - 요구수집-빠른설계-프로토타입 구축-고객평가-프로토타입 조정-구현-(반복)
    - 나선형 모형 : 점진적, 위험 관리
        - 계획수립-위험분석-개발및검증-고객평가-(반복)
    - 에자일 모형 : 특정 방법론X, 고객소통 중시, 짧은 개발주기 반복
        - 에자일 선언
            - 프로세스, 도구보다 개인과 상호작용에 가치를 둠
            - 방대한 문서보다 실행되는 SW에 가치를 둠
            - 계약 협상보다 고객 협업에 가치를 둠
            - 계획 따르기보다 변화에 반응에 가치를 둠
        - 스크럼
            - self-organizing, self-functional (스스로 팀 조직, 스스로 해결)
            - 구성원
                - 제품 책임자 (개발 의뢰자, 사용자)
                    - 제품 이해도가 높음
                    - 백로그(요구사항 우선순위) 작성, 주기적 갱신
                - 스크럼 마스터
                    - 조언자, 팀원 통제 X
                    - 일일 스크럼 회의 주관
                - 개발팀
                    - 이외 팀원들 (디자이너, 테스터 포함)
            - 개발 프로세스
                - 백로그
                - 스프린트 계획회의
                    - 단기 일정 수립
                - 스프린트
                    - 실제 개발기간
                - 일일 스크럼 회의
                    - 짧은 시간 진행사항 점검
                - 스프린트 검토 회의
                    - 제품이 요구사항 부합하는지 테스팅 수행
                    - 제품 사용자 참석
                - 스프린트 회고
                    - 규칙 준수 여부 및 개선점 확인
        - XP (익스트림 프로그래밍)
            - 고객 요구사항 유연한 대응
            - 고객 참여-개발 과정의 반복
            - 짧고 반복적인 개발주기
            - 릴리즈(부분적 완성) 기간을 짧게 반복
            - 소규모에 효과적
            - 핵심 가치
                - 의사소통, 단순성, 용기, 존중, 피드백
            - 개발 프로세스
                - 사용자 스토리 : 요구사항을 시나리오로 표현
                - 릴리즈 계획 수립
                - 스파이크 : 특정 기술 문제를 위한 간단한 프로그램 제작
                - 이터레이션 : 릴리즈보다 작은 단위
                - 승인 검사

## UML

- UML
    - 객체지향 모델링 언어
    - 사물
        - 종류 : 구조(클래스), 행동(상호작용), 그룹(요소를 그룹으로), 주해(부가 설명, 제약 등)
    - 관계
        - 연관 관계 Association : `<-` (관계를 갖는 방향, 양방향은 실선만)
            - ex. `teacher - student`
        - 집합 관계 Aggregation : `◇-` (part->whole 방향)
            - part와 whole은 독립적
            - ex. `computer ◇- printer`
        - 포함 관계 Composition : `◆-` (part->whole 방향)
            - part는 whole에 의존적
            - ex. `door ◆- key`
        - 일반화 관계 Generalizaion : `◁-` (일반화 방향)
            - 상속 (부모와 자식)
            - ex. 커피 ◁- 에스프레소
        - 의존 관계 Dependency : `<⋯` (영향을 주는쪽에서 받는 쪽으로)
            - 소유하지는 않으냐 영향을 주는 관계
            - ex. `할인률 <⋯ 등급` : 할인률 적용 위해 *등급*이 매개변수 활용
        - 실체화 관계 Realization : `◁⋯` (기능 방향으로)
            - ex. `걷기 ◁⋯ 사람`
    - 다이어그램
        - 구조적 다이어그램 (정적)
            - 클래스, 객체, 컴포넌트, 배치, 복합체 구조, 패키지
        - 행위 다이어그램 (동적)
            - 유스케이스, 순차, 커뮤니케이션, 상태, 활동, 상호작용, 타이핑
    - 스테레오 타입
        - UML의 기본 기능 외 추가 기능을 표현하기 위해 사용
        - `<< >>` 로 표현
- UML : 유스케이스 다이어그램 
    - tltmxpa, tltmxpa qjadnl

## CASE

- 개발 단계에서 사용하는 자동화 도구
- 개발 과정에서 사용되는 요구 분석, 설계, 구현, 검사 및 디버깅 과정을 전용 소프트웨어를 사용하여 자동화하는 작업
- 시스템 문서화, 명세화를 위한 GUI기능
- 표준화된 개발환경 구축, 문서자동화
- 작업과정 및 데이터 공유로 작업자 간 커뮤니케이션 증대

## Agile 에자일

- 절차와 도구보다 개인과 소통 중요시
- 고객과 피드백을 중요히
- 소프트웨어가 잘 실행되는데 가치
- 소프트웨어 배포 시차 최소화
- 특정 방법론이 아닌 소프트웨어를 빠르고 낭비없이 제작하기 위한 고객과 협업에 초점을 둠
- 특징 : 짧은 릴리즈와 반복, 점증적 설계, 사용자 참여, 문서 최소화, 비공식적인 커뮤니케이션 변화
- 선언문
    - 프로세스나 도구보다 개인과의 소통이 더 중요하다
    - 완벽한 문서보다 실행되는 소프트웨어가 더 중요하다
    - 계약 협상보다 고객과의 협업이 더 중요하다
    - 계획을 따르는 것보다 변경에 대한 응답이 더 중요하다

## 익스트림 프로그래밍 (Extreme Programming)

- 요구에 맞는 양질의 소프트웨어를 신속하게 제공하는 것을 목표로 한다
- 예측성보다는 적응성에 더 높은 가치
    - 요구사항을 미리 정의후 작업 진행 X
    - 요구사항이 변경되는 것을 적용하는 방식
- 고객의 참여와 개발 과정의 반복 극대화해 생산성 향상
- XP 핵심 가치
    - 소통 : 개발자, 관리자, 고객 간 원활한 소통 지향
    - 단순성 : 부가적 기능, 미사용 구조, 알고리즘 배제
    - 피드백 : 변화는 지속적 테스트와 통합, 반복적 결함 수정 등 빠르게 피드백
    - 용기 : 고객 요구사항 변화에 능동적 대처
    - 존중 : 개발 팀원간 상호 존중

## Use case 의 구성 요소 간의 관계

- 연관 관계 : 유스케이스와 액터간 상호작용이 있음을 나타냄
- 포함 관계 : 하나의 유스케이스가 다른 유스케이스의 실행을 전제로 할때 형성되는 관계
- 확장 관계 : 확장 기능 유스케이스와 확장 대상 유스케이스 사이 형성되는 관계
- 일반화 관계 : 유사한 유스케이스 또는 액터를 모아 추상화한 유스케이스/액터와 연결해 그룹을 만들어 이해도를 높이기 위한 관계
